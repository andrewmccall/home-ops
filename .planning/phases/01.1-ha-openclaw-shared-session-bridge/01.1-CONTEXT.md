# Phase 01.1: HA OpenClaw Shared Session Bridge - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 01.1 inserts a lightweight bridge between Home Assistant voice and OpenClaw so Home Assistant can keep using its UI-configured OpenAI conversation flow while the bridge injects the stable shared Ada/Andrew continuity metadata that current Home Assistant and OpenClaw surfaces do not provide directly. This phase is an unblocker for shared session continuity and transport compatibility; it does not add wake-word work, event forwarding, or the later approved Home Assistant action bridge.

</domain>

<decisions>
## Implementation Decisions

### Bridge contract
- **D-01:** The bridge should expose an OpenAI-compatible drop-in endpoint that the built-in Home Assistant OpenAI Conversation config entry can target through its existing base URL, API key, and model settings.
- **D-02:** The phase should preserve the Home Assistant UI/config-entry setup path rather than requiring a custom Home Assistant fork or bespoke operator workflow just to wire the conversation backend.

### Shared identity and continuity
- **D-03:** All Home Assistant voice turns in this first unblocker cut should use one durable Andrew/Ada shared identity so Home Assistant voice and Telegram continue the same Ada conversation instead of creating a separate HA-only session.
- **D-04:** The bridge, not Home Assistant, is responsible for injecting the stable OpenClaw continuity metadata needed to preserve that shared identity.

### Degraded-mode behavior
- **D-05:** If the bridge cannot provide the stable continuity metadata, voice may fall back to a stateless/shared-token path so the user still gets a response.
- **D-06:** That fallback must be surfaced clearly as degraded mode; silent continuity loss is not acceptable.

### LiteLLM cutover
- **D-07:** If the new bridge fully covers the current Home Assistant/OpenClaw proxy role, this phase should remove LiteLLM entirely rather than leave it running in parallel.

### the agent's Discretion
- Exact continuity mechanism (`user`, `x-openclaw-session-key`, or another supported OpenClaw-compatible field) as long as it preserves the single durable Andrew/Ada identity.
- Exact bridge runtime, namespace placement, and manifest layout as long as it remains a lightweight in-cluster service.
- Exact degraded-mode response shape in the bridge and Home Assistant-facing output, as long as the user can tell continuity is degraded.
- Exact LiteLLM removal sequence, health checks, and rollback safeguards.

</decisions>

## Existing Code Insights

### Reusable Assets
- `cluster/apps/openclaw/instance/ha-config-snippet.yaml`: Existing operator-facing setup notes for Home Assistant's OpenAI Conversation config entry and Ada assistant wiring.
- `cluster/apps/openclaw/instance/openclaw-instance.yaml`: Current OpenClaw gateway and session configuration that the bridge must call without assuming unsupported sender-mapping features.
- `cluster/apps/house/home-assistant/home-assistant.yaml`: Existing Home Assistant deployment and voice plumbing that this phase should reuse rather than rebuild.
- `cluster/apps/openclaw/litellm/litellm.yaml`: Existing OpenAI-compatible proxy shape proving an adapter layer can sit in front of OpenClaw.

### Established Patterns
- Home Assistant voice infrastructure is already deployed; the missing piece is conversation backend continuity, not STT/TTS.
- OpenClaw configuration lives in `spec.config.raw`, and the OpenAI-compatible endpoint already exists behind the shared gateway token.
- The repo already uses Kubernetes manifests and service URLs for internal integration rather than relying on public ingress paths for in-cluster traffic.
- The previous HA path depended on an OpenAI-compatible surface, so a drop-in bridge aligns with current operator expectations better than a custom Home Assistant component.

### Integration Points
- Home Assistant OpenAI Conversation config-entry base URL/API key/model configuration.
- New bridge service sitting between Home Assistant and `home-ops-openclaw.openclaw.svc`.
- OpenClaw continuity metadata injection for the single shared Andrew/Ada session.
- LiteLLM deployment/manifests as the proxy surface to replace and potentially remove.

</code_context>

<specifics>
## Specific Ideas

- Keep Home Assistant setup looking normal from the UI: point it at a base URL and token, and let the bridge handle the OpenClaw-specific continuity details.
- Preserve one shared Andrew/Ada conversational identity across Home Assistant voice and Telegram in this first unblocker cut.
- If continuity has to degrade, make that visible instead of pretending the user is still in the same session.
- If the bridge covers the current proxy job, remove LiteLLM rather than carrying duplicate proxy layers.

</specifics>

<deferred>
## Deferred Ideas

- Multi-user or per-device identity/session policies beyond the single Andrew/Ada shared identity.
- Any Home Assistant custom integration or fork work that goes beyond the OpenAI-compatible drop-in bridge approach.
- Wake-word, event-bridge, and approved action-surface work, which remain in later phases.

</deferred>
