# Phase 01.1: HA OpenClaw Shared Session Bridge - Research

**Researched:** 2026-04-13
**Domain:** Home Assistant `openai_conversation` compatibility bridge for OpenClaw shared session continuity
**Confidence:** High

## Summary

Home Assistant `2026.3.4` no longer behaves like a simple Chat Completions client for `openai_conversation`. The current integration validates with `GET /v1/models` and sends requests through the OpenAI **Responses API**, so a bridge planned around `/v1/chat/completions` alone is the wrong target.

OpenClaw already has the needed primitives: `/v1/models`, native `/v1/responses`, `x-openclaw-session-key`, and `x-openclaw-message-channel`. The current live cluster has chat completions enabled, but responses support still needs to be enabled in the OpenClaw gateway config. The shortest safe implementation path is therefore:

1. Enable OpenClaw responses support.
2. Put a very small OpenAI-compatible bridge in front of OpenClaw.
3. Have that bridge inject a fixed shared session key and channel metadata for Home Assistant voice.
4. Preserve explicit degraded-mode behavior if continuity injection fails.
5. Move ingress/TLS ownership before removing LiteLLM.

LiteLLM is already not the live Home Assistant request path; the current `api.openai.com` spoof ingress routes directly to OpenClaw. That means LiteLLM removal is cleanup work, but the ingress/TLS assets currently living under `cluster/apps/openclaw/litellm/` must be preserved before removing that directory.

## Key Findings

### 1. The bridge must be Responses-compatible

The Home Assistant integration now depends on:

- `GET /v1/models`
- `POST /v1/responses`
- streaming Responses behavior

Planning a chat-completions-only bridge would recreate the same mismatch that blocked Phase 1.

### 2. The bridge should mutate requests, not translate protocols

The best bridge is a thin passthrough that:

- accepts the HA-facing OpenAI-compatible request
- authenticates HA with its own bridge API key
- forwards upstream with the shared OpenClaw gateway token
- injects:
  - `x-openclaw-session-key: <fixed-shared-key>`
  - `x-openclaw-message-channel: home_assistant_voice`

This keeps the bridge small and avoids fragile Responses-to-Chat-Completions translation.

### 3. Preserve the existing HA-facing spoof path

The repo already has an `api.openai.com` spoof path via:

- custom CA in Home Assistant
- host aliasing in the HA deployment
- ingress/TLS assets under the LiteLLM folder

The bridge should likely take over that path rather than forcing a different HA integration approach.

### 4. Use a dedicated bridge-facing key

Home Assistant should authenticate to the bridge with its own key, while the upstream OpenClaw gateway token remains server-side in the bridge secret. This narrows exposure compared to giving HA the full OpenClaw gateway token directly.

### 5. Degraded mode is required, but it must be explicit

If continuity injection fails, the bridge may proxy statelessly so voice still works, but it must surface degraded mode clearly instead of silently creating a separate session.

## Recommended Structure

```text
cluster/apps/openclaw/
├── bridge/
│   ├── bridge.yaml
│   ├── bridge-secret.sops.yaml
│   ├── openai-proxy-ingress.yaml
│   ├── openai-proxy-tls-secret.yaml
│   └── kustomization.yaml
├── instance/
│   └── openclaw-instance.yaml   # enable gateway.http.endpoints.responses
└── litellm/                     # removable after ingress/TLS ownership moves
```

This matches the repo's existing Helm/Kubernetes packaging patterns and gives the bridge a clean ownership boundary.

## Anti-Patterns to Avoid

- Building only `/v1/chat/completions`
- Translating Responses requests into Chat Completions instead of forwarding to OpenClaw native Responses
- Deleting `cluster/apps/openclaw/litellm/` before ingress/TLS assets move elsewhere
- Assuming HA can just be pointed at a completely different custom integration path
- Treating the shared OpenClaw gateway token as a narrow-scoped HA credential

## Operational Risks

1. **Responses support disabled upstream**  
   `openclaw-instance.yaml` must enable `gateway.http.endpoints.responses`.

2. **Ingress/IP mismatch**  
   There is a live mismatch between the HA hostAlias target IP and the current ingress-advertised IP, so planning should include explicit validation before cutover.

3. **NetworkPolicy gaps**  
   If the bridge runs in a different namespace or with different labels than current allowed sources, OpenClaw ingress policy may need to change.

4. **Silent continuity loss**  
   Degraded fallback must be visible in behavior and logs.
